Problem Statement & System Architecture

Challenge: Traditional best-effort networks cannot differentiate between latency-sensitive traffic (VoIP, video) and bulk transfers, causing congestion and poor QoE.

SDN Architecture Solution

Controller: Ryu (Python) + QoS Application
↓ OpenFlow 1.3 ↓
Control: Classify Traffic → Decide Policy → Install Rules
↓ FlowMod | MeterMod ↓
Data Plane: Open vSwitch - Leaf-Spine (1 Core + 2 Leaf + 4 Hosts)
↓ 50 Mbps Bottleneck ↓
Testing: iperf, ping
Key: Centralized intelligence + Programmable forwarding = Dynamic QoS
Ref: ONF White Paper 2012; Lantz et al., HotNets-IX 2010

QoS Enforcement Mechanisms

Priority Queues (Week 2)
  • Q0 (High): 10-30 Mbps, Priority 100
  • Q1 (Normal): 5-20 Mbps, Priority 50
  • Q2 (Low): 1-50 Mbps, Priority 10

→ Controls forwarding ORDER

Meter Tables (Week 3)
  • High: 10 Mbps guaranteed
  • Normal: 5 Mbps guaranteed
  • Excess → Remark/Drop

→ Controls sending RATE

Ref: Ryu Docs; OpenFlow 1.3 Spec; Dhrub et al., ECCE 2019

End-to-End Packet Processing Workflow

1. Packet arrives → No flow match → PacketIn to Controller
↓
2. Parse headers (Eth/IP/TCP/UDP) → Extract ports/DSCP
↓
3. Classify traffic → Assign priority class (HIGH/NORMAL/LOW)
↓
4. Policy engine → Determine Queue ID + Meter ID
↓
5. Install Meter (OFPMeterMod) → Rate limit: 10 Mbps CIR, 15 Mbps PIR
↓
6. Install Flow (OFPFlowMod) → Match + Actions (SET_QUEUE, OUTPUT)
↓
7. Subsequent packets → Data plane only (0 controller overhead) ✓
Critical Insight: First packet = controller (setup), Rest = hardware speed (scalable)
Flow setup ~2-5ms once, then wire-speed forwarding
